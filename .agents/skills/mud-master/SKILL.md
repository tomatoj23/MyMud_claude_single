# MUD Master

将 `mud-master` 作为本项目唯一的本地 Skill 使用。在同一上下文内完成规划、实现、验证和审查，不再切换到其他项目 Skill。

## 先判定信息来源

按下面顺序取证，不要混淆优先级：

1. **当前进度与阶段完成度**：先读仓库根目录的 `TODO.md`、`docs/tech_debt_report.md`、`DEVELOPMENT_PLAN.md`
2. **专项迁移或兼容性改造**：如果任务涉及迁移、兼容性、专项回归，再补读 `MIGRATION_LOG.md`
3. **当前真实实现**：再读 `src/`、`tests/`、相关运行配置和当前文档
4. **设计蓝图与未来阶段目标**：最后再读 `references/` 下的阶段文档和专题设计文档

### 根目录文档冲突处理

- `TODO.md` 是当前执行状态和近期推进情况的最高优先级来源。
- `docs/tech_debt_report.md` 用于确认测试覆盖、技术债务结论，以及阶段四是否已经启动。
- `DEVELOPMENT_PLAN.md` 用于总体里程碑和原始阶段规划；如果与 `TODO.md` 冲突，优先级低于 `TODO.md`。
- `MIGRATION_LOG.md` 只在迁移、兼容性、专项改造任务中作为补充证据，不单独决定全局阶段完成度。
- 当根目录文档冲突时，优先使用“日期更近且任务更具体”的文档，并在回复中点明冲突来源。
- 如果文档和当前代码也发生冲突，以实际代码、测试和可运行验证结果为最终兜底，并指出文档待更新。

## 当前状态基线

先以 `TODO.md` 与 `docs/tech_debt_report.md` 在 `2026-02-26` 附近记录的状态作为默认基线，再在回答前复核：

- 阶段一：已完成
- 阶段二：已完成
- 阶段三：已完成
- 阶段四：进行中
- 阶段五：待开始
- 阶段六：待开始

回答“当前做到哪一步”“下一阶段是什么”“哪些模块还没做”之前，必须先读根目录进度文档；如果 `DEVELOPMENT_PLAN.md` 仍显示阶段四“待开始”，而 `TODO.md` / `docs/tech_debt_report.md` 显示“进行中”，以较新的执行文档为准，并在回复中说明差异。

## 路径处理规则

处理 `references/` 中的 `# src/...` 路径时，遵守以下规则：

- 如果路径在当前仓库中存在，把它当作当前实现锚点。
- 如果路径不存在且对应阶段已经完成，把它视为历史蓝图、计划中的模块拆分，先映射到现有相邻文件，再决定是否真的需要新建文件。
- 如果路径不存在且对应阶段尚未开始或仅在进行中，把它视为允许的计划路径。
- 不要因为 reference 写了某个文件名，就直接新建同名文件；先核对当前目录树、调用点和测试覆盖范围。

## 判断任务类型

先识别用户当前最需要哪一种能力：

- **规划**：拆解阶段目标、模块顺序、依赖关系、验收标准
- **实现**：阅读现有代码，直接修改工作区文件，并补齐测试或文档
- **审查**：检查代码、文档、测试和配置，按严重程度列出问题和风险
- **混合任务**：当用户明确要求“规划并实现”“实现并审查”“全面检查项目”时，再串联多个步骤

## 通用工作原则

1. 先检查仓库现状，再决定读取哪些 reference 文件。
2. 只加载与当前任务直接相关的 `references/` 文档，避免一次性读取全部资料。
3. 当 reference 与仓库代码不一致时，以仓库现状为准，并明确标注“这是当前实现”还是“这是未来蓝图”。
4. 对可直接落地的任务，优先修改文件并执行验证；不要只返回大段示例代码。
5. 对审查任务，先给 findings，再给结论；不要用固定模板伪造“已通过”。
6. 无法运行的测试或检查要明确说明限制，不要假设成功。

## 规划流程

用户要求阶段规划、任务拆分、进度检查或验收准备时：

1. 先读取根目录的进度文档，确认当前阶段与完成度。
2. 再读取对应的 `references/phase*.md`，提炼阶段目标、依赖、验收项和风险。
3. 如果 phase 文档与根目录状态冲突，以根目录进度文档为准，并在回复中指出 phase 文档属于规划蓝图或历史设计。
4. 输出面向当前仓库状态的计划，而不是机械复述开发蓝图。

## 实现流程

用户要求实现、修复、重构、补测试或补文档时：

1. 先阅读相关源码、测试和根目录进度文档。
2. 再读取最小必要的 reference 文档。
3. 先完成“蓝图路径 → 当前文件”的映射，再开始改代码。
4. 对阶段一到三这类已完成阶段，优先在现有文件上增量修改，不要重新按蓝图起新骨架。
5. 对阶段四到六这类进行中或未开始阶段，可以把 reference 当作规划蓝图，但仍需先确认是否真的要新建文件。
6. 同步补充测试、配置或文档，确保改动能落地。
7. 运行可用验证，并在结果中明确写出已验证项和未验证项。

## 审查流程

用户要求审查、验收、找问题、评估风险时：

1. 先检查实际代码、文档、测试和配置。
2. 再按需读取 `references/checklists.md`、`references/tests.md`、`references/acceptance.md`。
3. 同时对照根目录进度文档，确认“这是未实现内容”还是“这是与计划偏离的问题”。
4. 以“发现的问题”为主输出，按严重程度排序。
5. 为每个问题给出具体文件位置和修正方向。
6. 如果没有发现问题，要明确说明，并补充剩余风险或测试缺口。

## 标准验证入口

优先使用仓库已经定义好的验证入口：

1. `make lint` / `make test` / `make check`
2. `python -m tools.dev lint` / `python -m tools.dev test` / `python -m tools.dev check`
3. 如果当前 shell 没有 `python` 命令，改用 `py -m tools.dev ...`
4. 只有在标准入口不可用、依赖缺失或任务非常局部时，才退化为更小粒度的导入检查、单文件测试或临时命令

- 阶段一到三的 reference 文档里，可能保留设计期的目录结构和拆分模块示例。
- 这些示例对“理解设计意图”有用，但不一定等于当前仓库的真实文件布局。

## 项目约束

- Python 3.11+
- 数据库：SQLite + SQLAlchemy asyncio + aiosqlite
- 异步：`asyncio` / `async` / `await`
- GUI：PySide6 + qasync
- 序列化：MessagePack，避免 Pickle
- 代码尽量保持完整类型注解
- 避免阻塞事件循环
- 新增或修复功能时优先补对应测试

## 输出要求

### 规划输出

- 给出阶段建议、依赖关系、推荐顺序、验收要点
- 明确哪些结论来自根目录进度文档，哪些来自 reference 设计文档

### 实现输出

- 说明实际修改了哪些文件
- 概括关键实现点，而不是粘贴整份代码
- 报告验证结果、未验证项和剩余风险

### 审查输出

- 先列 findings，按严重程度排序
- 给出文件路径和行号
- 区分“未实现的未来阶段内容”和“当前阶段的真实问题”
- 如未运行测试或验证，要明确写出
