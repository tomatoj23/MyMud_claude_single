## 高优先级债务

### 1. 单例模式状态污染

**位置**: `src/utils/config.py`, `src/engine/core/messages.py`

**问题描述**:
```python
# ConfigManager 单例
class ConfigManager:
    _instance: ConfigManager | None = None
    
    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
        return cls._instance
```

**风险**:
- 测试之间相互影响
- `test_detect_environment_frozen` 偶尔失败
- 无法并行运行测试

**影响**: 🔴 高 - 影响测试稳定性

**建议方案**:
- 添加 `reset()` 方法用于测试清理
- 或改用依赖注入替代单例

---

### 2. 异步代码中的同步调用风险

**位置**: `src/engine/core/messages.py:57`

**问题描述**:
```python
class Message:
    def __init__(self, ...):
        self.timestamp = asyncio.get_event_loop().time()
```

**风险**:
- 在没有事件循环的线程中调用会抛出 `RuntimeError`
- 测试 `test_msg_no_caller` 曾因这个问题失败

**影响**: 🔴 高 - 可能导致运行时崩溃

**建议方案**:
- 使用 `time.time()` 替代
- 或延迟获取时间戳直到需要时

---

### 3. 战斗系统属性名不一致

**位置**: `src/game/combat/core.py`

**问题描述**:
- `Character` 类使用 `key` 作为名称
- 战斗系统原使用 `character.name`（不存在）
- 已临时修复为 `character.key`

**风险**:
- 代码可读性降低
- `key` 作为标识符，`name` 作为显示名更合理

**影响**: 🟡 中 - 技术债务累积

**建议方案**:
- 为 `Character` 添加 `name` 属性
- 或明确区分 `key`（唯一ID）和 `name`（显示名）

## 架构层面债务

### 23. 模块耦合

**分析**:
```
┌─────────────┐     ┌─────────────┐
│   combat    │◀───▶│   npc       │
│             │     │             │
│  直接依赖    │     │  直接依赖    │
│  Character  │     │  Character  │
└─────────────┘     └─────────────┘
         │                │
         └───────┬────────┘
                 ▼
          ┌─────────────┐
          │  Character  │
          │  职责过多    │
          └─────────────┘
```

**问题**:
- `Character` 类职责过多（角色、装备、武学）
- 通过 Mixin 组合，但耦合度仍然较高

**影响**: 🟡 中 - 长期维护

**建议方案**:
- 考虑使用组合替代继承
- 组件化设计（ECS架构）

---

### 24. 数据库设计限制

**问题**:
- 属性存储为JSON，查询能力受限
- 无法直接查询属性内的字段
- 需要全表扫描

**影响**: 🟡 中 - 扩展性

**建议方案**:
- 对常用查询字段建立索引
- 考虑使用 JSON 索引（SQLite 3.38+）
- 或规范化部分常用属性
